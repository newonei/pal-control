# 玩家门户部署手册

## 1. 目标拓扑

```text
Internet
   |
   |  TCP 80/443, optional UDP 443
   v
Caddy (public TLS endpoint)
   |-- static files --> apps/player-web/dist
   `-- /api/v1/player/* --> http://127.0.0.1:5180

Control API, admin console, Palworld REST, PalDefender and RCON remain local-only.
```

模板使用同源部署，因此浏览器不需要跨域直连 Control API。所有其他 `/api` 路径由 Caddy 直接返回 `404`，不会落入 SPA，也不会到达后端。

## 2. 上线前置条件

- 已拥有一个专用玩家门户子域名；不要复用 RCON、游戏 Query 或运营控制台的地址。
- `apps/player-web` 已实现生产构建，且 `dist/index.html` 存在。
- 玩家 API 只信任服务端会话中的平台身份；不能接受前端自行提交的任意 `userId` 作为授权依据。
- Control API 实际监听地址为 `127.0.0.1:5180`，从另一台机器连接失败。
- Caddy 使用独立低权限 Windows 服务账户；该账户不能读取 `appsettings.Local.json`、RCON 密码或 PalDefender token。
- Windows 防火墙与路由器只为门户增加 `80/TCP`、`443/TCP`，需要 HTTP/3 时才增加 `443/UDP`。

## 3. 构建和静态文件准备

在代码审查、依赖锁文件检查和前端测试通过后，使用项目定义的玩家站点构建命令。例如工作区注册完成后可执行：

```powershell
npm ci
npm run build --workspace apps/player-web
```

发布前检查：

```powershell
Test-Path .\apps\player-web\dist\index.html
Get-ChildItem .\apps\player-web\dist -Recurse -Force
```

不得把以下内容放入 `dist`：`.env`、源码仓库、私钥、source map 中的秘密、运营控制台构建、服务器配置或真实玩家数据。Caddy 模板会对任何包含点号开头路径段的请求直接返回 `404`，但这只是第二道防线，不能替代发布物扫描。生产环境宜把已验收的 `dist` 复制到只读的版本目录，再原子切换 `PLAYER_PORTAL_ROOT`；不要在 Caddy 正在读取的目录内逐文件覆盖。

## 4. DNS

1. 创建专用子域的 `A` 记录，指向 Caddy 主机的公网 IPv4。
2. 只有在 IPv6 路由和防火墙都配置完成时才创建 `AAAA`。错误的 `AAAA` 会造成部分用户无法连接。
3. 等待公共递归 DNS 能解析到预期地址，再启动正式证书申请。
4. 不要为 `5180`、`8212`、`17993` 或 `25575` 创建 SRV、端口转发或独立公网主机名。

## 5. Caddy 变量与配置

复制 [Caddyfile](../../deploy/player-portal/Caddyfile) 到受管配置目录，例如 `C:\ProgramData\Caddy\Caddyfile`。为 Caddy 服务进程设置：

```text
PLAYER_PORTAL_DOMAIN=<your-public-player-domain>
PLAYER_PORTAL_ROOT=C:/PalControl/apps/player-web/dist
PLAYER_PORTAL_ACCESS_LOG=C:/ProgramData/Caddy/logs/player-portal-access.json
```

尖括号值必须替换；不要把真实值提交到仓库。`player-portal.env.example` 不会被 Caddy 自动读取。使用 WinSW 时可在本机服务配置中设置非秘密环境变量；使用 `sc.exe` 时可设置系统级变量并重启服务。配置变更前先在与服务相同的环境和身份下运行：

```powershell
caddy fmt --diff --config C:\ProgramData\Caddy\Caddyfile
caddy validate --config C:\ProgramData\Caddy\Caddyfile --adapter caddyfile
caddy adapt --config C:\ProgramData\Caddy\Caddyfile --adapter caddyfile --pretty
```

`adapt` 输出中必须只有一个后端：`127.0.0.1:5180`；匹配器必须只覆盖 `/api/v1/player/*`。

Control API 的本机配置还必须显式启用玩家门户，并把唯一公开站点加入来源白名单：

```json
{
  "PlayerPortal": {
    "Enabled": true,
    "CookieSecure": true,
    "AllowedOrigins": [
      "https://shop.example.com"
    ]
  }
}
```

`AllowedOrigins` 必须与浏览器地址的协议、域名和端口完全一致，不能写 `*`。公网部署必须保持 `CookieSecure=true`；仅回环地址上的 HTTP 开发调试才允许设为 `false`，并且程序会拒绝在非开发模式下以不安全 Cookie 启动。验证码和会话保存在 Control API 进程内，服务重启会使验证码及玩家登录会话失效，但不会影响钱包、订单或账本。

Player API 在受信任的 `X-Forwarded-For` 处理之后按真实客户端 IP 使用固定窗口限流：默认认证路径每分钟 30 次、`/me` 路径每分钟 240 次；整个玩家命名空间最多同时处理 64 个请求且不排队。Caddy 与 Kestrel 都把玩家请求体限制为 16 KiB。`429` 的 `Retry-After` 必须保留给浏览器，不能在上层代理中改写。修改这些阈值前要先做容量测试；标准 Caddy 配置仍不能代替 CDN/WAF 的分布式 DDoS 防护。

## 6. TLS 与服务启动

Caddy 对配置中的公共域名默认启用自动 HTTPS。证书签发需要：

- DNS 已指向本机；
- 公网 `80/TCP` 和 `443/TCP` 能到达 Caddy；
- Caddy 可以绑定这些端口并向 ACME 服务发起出站连接；
- 系统时间准确；
- Caddy 服务账户的数据目录持久且可写。

首次启动前确认日志目录存在且只有服务账户和管理员可写。将 Caddy 安装为 Windows 服务并设置自动启动与失败重启。证书和 ACME 状态必须存放在持久目录；清空该目录会触发重新签发并可能撞上 CA 限额。

不要在模板中手工降低 TLS 版本、关闭证书校验或使用 `tls_insecure_skip_verify`。若 `80` 无法开放，可另行采用受支持的 DNS challenge 插件；DNS API token 必须放在服务密钥存储中，不能写进 Caddyfile 或仓库。

## 7. 发布、更新与回滚

推荐流程：

1. 在临时目录完成前端构建和安全扫描。
2. 校验资源哈希、`index.html`、CSP 兼容性和 API 基址。
3. 把完整构建复制到新的只读版本目录。
4. 修改本机服务变量指向新目录，执行 `caddy validate`。
5. 使用 `caddy reload` 无中断加载配置。
6. 执行 [验收清单](04-验收清单.md)。
7. 保留上一个无漏洞版本；出现问题时把根目录切回旧版本并 reload。

回滚只回滚静态文件或代理配置，不能回滚数据库账本。任何玩家购买或货币写入仍按 Control API 的幂等与对账流程处理。

## 8. 运维

- 监控证书续期、HTTP `5xx`、异常 `404`、登录失败率、磁盘和日志滚动。
- Caddy 访问日志使用 JSON，并保留有限数量和天数。默认不要记录凭据；令牌也不得放在 URL 查询参数中。
- 变更 Caddyfile 后使用 reload，不用停服替换；变更日志输出文件选项时按 Caddy 文档安排服务重启。
- 每次 Control API 或玩家路由升级后，重新验证“其他 `/api` 为 404”和外网无法直连内部端口。
