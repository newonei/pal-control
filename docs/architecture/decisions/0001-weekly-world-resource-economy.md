# ADR-0001：采用周世界资源经济服

- 状态：已接受
- 决策日期：2026-07-13
- 决策人：项目维护者
- 关联路线图：[玩法 TODO](../../../TODO.md)

## 背景

当前实现已经能够在常驻周世界中读取玩家位置和允许容器，为白名单资源生成限时报价，确认扣物后发放周战备券。它没有独立的行动开始/结束生命周期，也没有能够证明资源获得时间和来源的权威事件。

此前有两个候选方向：

- 方案 A：周世界中的任意白名单资源都可出售，核心是资源生产、流通、回收和战备消费；
- 方案 B：只有明确开始行动后获得的新增战利品可结算，需要行动状态机和不可伪造的战利品来源。

## 决策

采用 **方案 A：周世界资源经济服**，作为当前产品基线和发布目标。

核心循环是：

```text
常驻周世界采集/战斗/生产
  -> 到达资源兑换点
  -> 扫描允许容器中的白名单资源
  -> 确认整单报价并原子扣物
  -> 获得周战备券
  -> 在战备商城消费
  -> 周换档并结算排名/奖励
```

具体规则如下：

1. 只要资源位于当前周世界、属于生效回收目录且位于允许扫描的容器，就可以出售；不要求证明采集时间、掉落来源或所属行动。
2. 当前允许容器沿用 `Items`、`Food` 和 `DropSlot`。目录外物品默认不可出售。
3. 玩家必须位于开放的资源兑换点，服务端完成位置校验后才能生成报价。
4. 报价包含扫描时所有合格资源的全部数量。玩家可以取消整单，但当前版本不能在同一报价中选择保留部分合格资源；选择性出售属于远期体验优化。
5. 报价有效期沿用当前配置（默认 30 秒）。确认后必须先以可归因、幂等、全成或全败的方式扣物，才能唯一入账。
6. 报价过期或取消时不扣物、不入账；结果不能证明、部分扣物或响应丢失时进入人工核对，禁止盲目重试或重复入账。
7. 玩家死亡、断线、放弃游戏或在世界内停留多久都不是资源兑换的经济状态，也不产生“行动成功/失败”。这些事件仍可用于服务器运维、任务或风控，但不得伪造成兑换结果。
8. 周换档前必须排空或人工处置未终结的资源兑换、订单和 `uncertain` 记录，并完成世界与经济状态备份。

## 为什么不采用方案 B

方案 B 当前不是对已有功能的增量修补，而是另一种产品：它要求独立行动生命周期、携入物规则、死亡/断线结果、权威战利品来源和新的作弊对抗模型。现有 PalDefender/Native 接口尚不能可靠证明同一 ItemID 的获得来源；仅比较行动前后背包数量也无法排除基地箱取回、交易、制作、预放物和丢弃再拾取。

更重要的是，当前目标乐趣已经明确为周世界中的资源经营和循环消费，而不是逐局搜打撤。先完成方案 A 能复用现有闭环，把工程投入集中在原子扣物、结算幂等、身份、运营安全和经济平衡上。

方案 B 不作为本版本 P0/P1 发布门槛，也不需要为证明“不适用”实现 Native 战利品归因 spike。未来只有在重新确认产品定位并证明权威来源能力后，才能通过新的 ADR 重新立项；不得直接复用资源兑换 settlement 充当行动记录。

## 术语与兼容性

面向玩家和运营使用以下术语：

| 使用 | 不再作为玩法术语使用 |
| --- | --- |
| 周世界资源经济服 | 行动制搜打撤 |
| 资源兑换点 | 撤离点 |
| 资源报价 / 资源兑换 | 战利品撤离 |
| 兑换记录 / 结算记录 | 行动记录 |
| 已兑换 / 兑换失败 / 待人工核对 | 撤离成功 / 行动失败 |

现有 API 和代码中的 `Extraction*`、`run`、`extractedItemCount`、`extractedValue` 以及状态值 `extracted` 是兼容性遗留名。本阶段不为改名破坏客户端或历史数据；对外展示必须按资源兑换语义解释。若以后清理这些字段，应使用版本化 API、数据迁移和明确的兼容窗口。

资源兑换状态的玩家端映射为：

| 兼容状态值 | 展示 |
| --- | --- |
| `preparing` | 待确认 |
| `deployed` | 结算中 |
| `extracted` / `settled` | 已兑换 |
| `failed` | 兑换失败 |
| `uncertain` | 待人工核对 |
| `cancelled` | 已取消或已过期 |

`run` 只代表一笔资源兑换 settlement，不代表玩家从进入世界到离开的完整过程。技术结算失败也不得解释为玩家玩法失败。

## 排行榜和赛季统计

方案 A 只发布能够从结算账本复算的指标：

- 周资源兑换总价值；
- 按 ItemID/资源类别汇总的有效兑换数量和价值；
- 经冻结规则计算的任务积分；
- 商城消费、订单完成等经济指标（仅在玩法需要且不泄露敏感数据时展示）。

不发布“行动次数”“撤离成功率”“死亡率”或“本局战利品价值”排行榜。`successfulRuns`、`failedRuns` 等现有兼容字段在迁移前只能视为技术结算计数，不能用于玩家竞技排名或周奖励。

排行榜必须基于赛季结束时的冻结快照，排除被封禁/冻结账户，固定同分规则和迟到结算截止时间；奖励写入幂等账本。

## 结果

正面影响：

- 与当前实现和真实可验证能力一致，减少一次尚无可靠技术基础的底层改版；
- 玩家可以自由采集、生产、交易和囤积，再选择合适时机兑换，强化周世界资源经营；
- 当前开发重点可以集中在经济副作用的安全性、稳定性、内容量和运营能力。

代价与风险：

- 无法宣称“只出售本次行动战利品”，基地生产、交易所得和此前囤积的白名单资源同样可售；
- 宏、自动化农场、玩家交易和制作链可能放大通胀，需要回收目录、价格、限额、异常账户检测和经济仿真共同约束；
- “撤离”相关内部命名会在一段时间内与产品术语不一致，需要文档和 UI 明确兼容语义；
- 默认整单出售可能误售玩家希望保留的材料，报价必须逐项、逐数量展示并要求显式确认；选择性出售成熟后再单独设计。

## 重新评审触发条件

以下任一条件出现时，应新建 ADR，而不是静默改变本决策：

- 产品明确转向逐局搜打撤，并把死亡、携入物和成功率作为核心体验；
- 固定游戏版本上出现可重复验证、不可伪造的物品来源事件或托管战利品容器；
- 资源经济无法通过限额、价格和风控控制通胀，且需要按来源限制出售；
- 计划改变“整单出售全部合格资源”或允许跨世界/跨服资源兑换。
