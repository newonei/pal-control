# 幻兽商域玩法 TODO

> 审计基线：2026-07-13，`main` 分支。本文只描述当前仓库能够证明的能力，以及从“本机开发服原型”走到“可长期运营的搜打撤玩法”仍需完成的工作。设计文档、单次手工演示或页面截图不等于功能完成。

## 当前结论

当前玩法更准确的定位是：

> **Palworld 常驻周世界 + 网页战备商城 + 固定撤离点资源兑换系统。**

它已经具备一条本机开发服可运行的经济闭环，但还不是严格意义上的行动制搜打撤：

- 没有真正的“开始行动、行动中、死亡、超时、放弃、成功撤离”生命周期；
- 当前 `run` 在生成撤离报价时才创建，本质是结算记录，不是一局行动；
- 没有权威战利品来源事件或托管容器，背包数量差无法区分本局战利品与此前囤积、交易或生产的资源；
- 死亡掉落规则没有进入经济安全门禁；
- 公开经济测试所需的 Native 原子扣物、管理员 RBAC、平台身份绑定、故障注入、恢复演练和监控尚未闭环。

因此，现阶段应继续标记为 **仅限可信本机/好友开发服的原型**，不能直接作为公网经济服开放。

## 当前实际可玩流程

1. 玩家输入平台 UserId/SteamID64，在线角色通过游戏内私信收到 8 位验证码。
2. 验证成功后建立玩家门户会话；首次访问创建跨周账户，并按开发配置发放 1000 商域币和 300 战备券。
3. 玩家浏览 5 个硬编码战备商品；商品基础价格每天按确定性种子调整到 90%–110%。
4. 购买时先扣数据库钱包，再由 PalDefender 发放物品；确定失败退款，结果无法证明时进入 `uncertain`。
5. 玩家回到常驻 Palworld 周世界自行采集、战斗和拾取资源。
6. 玩家门户地图每 5 秒更新位置；默认配置只有 1 个开发服撤离点。
7. 进入撤离区后，服务端间隔 2 秒进行两次位置采样。
8. 系统读取 `Items`、`Food`、`DropSlot`，对约 21 种白名单资源的**全部数量**生成 30 秒报价；当前不能选择只出售一部分。
9. 玩家确认后由 RCON `/delitems` 扣物，再用 PalDefender REST 回读总量。
10. 只有回读证明目标总量按报价减少后，系统才唯一增加战备券；无法证明则冻结并等待人工对账。
11. 玩家使用战备券继续购买补给，形成“采集/战斗 -> 资源出售 -> 战备采购”的循环。
12. 周档结束后由管理员运行换档脚本，保存旧世界、启动新世界并提交新赛季。

这条流程已经证明了基础交易链，但目前真实验收证据仍主要是一次商城发货和一次 5 个皮革撤离，尚未证明完整周档、并发和故障恢复。

## 必须先冻结的玩法规则

在继续扩商品、任务和排行榜前，先确认玩法定位：

- 方案 A：本周世界中的任意白名单资源都可出售，定位为“周世界资源经济服”。这最接近当前已实现的 MVP，只需把现有兑换结算做安全、稳定；
- 方案 B：只有玩家明确开始行动后获得的新增战利品可结算，定位为“行动制搜打撤”。这是一次产品和底层集成改版，不是对当前结算记录补几个状态。

**产品建议采用方案 B**：保留“每周一个常驻世界”，但在世界内增加逻辑行动会话，不为每局单独启动服务器。该建议仍须经过 P0-00 的可行性验证和 ADR 决策，不能把未被证明的战利品归因当成既定能力。

若选择 A，P0-01、P0-03 和“行动成功率”排行榜不适用；系统只保留真实的兑换结算状态和技术指标。若选择 B，必须实现以下逻辑行动状态：

```text
preparing -> cancelled | deployed
deployed -> extracting | reconnecting | died | abandoned | timed_out | terminated_by_admin
reconnecting -> deployed | timed_out | terminated_by_admin
extracting -> deployed                 # 报价取消/过期，且确认未扣物
extracting -> died | reconnecting | timed_out | terminated_by_admin  # consume 尚未派发，报价同时失效
extracting -> resolution_pending       # 已提交结算
resolution_pending -> extracted        # 仅 settlement=settled
resolution_pending -> deployed         # 明确未扣物、未入账
resolution_pending -> manual_review    # uncertain 或部分扣物
manual_review -> extracted | deployed | terminated_by_admin
```

撤离扣物本身使用独立的 settlement saga：

```text
quoted -> expired | cancelled | consuming
consuming -> removed | definitively_failed | uncertain
uncertain -> removed | definitively_failed | manual_review
manual_review -> removed | definitively_failed
removed -> credited -> settled
```

只有 `settled` 能把行动推进为 `extracted`；技术 `failed/uncertain` 不得记为玩家死亡或玩法失败。部分扣物不得进入普通失败路径；`reconnecting`、`extracting`、`resolution_pending` 和 `manual_review` 期间禁止开始新行动。死亡、断线、超时和世界切换必须来自权威游戏服务端/Native 事件，不能用网页是否在线推断。consume 一旦 dispatched，行动必须先进入 `resolution_pending`；此后到达的死亡/断线事件只追加事实，不能覆盖经济结算结果，直到 settlement 获得最终证据。存在任何未终结 settlement 时禁止换档或切换世界。

## 优先级定义

- **P0**：公开测试或对抗性多人测试前的硬门槛；任一未完成都必须保持经济玩法关闭或仅限可信开发服。
- **P1**：形成有内容深度、能连续运行完整周档的 MVP 所需工作。
- **P2**：在核心闭环连续稳定两个周档后再做的扩展。

标注 `[仅方案 B]` 的条目只在 P0-00 选定方案 B 后成为发布门槛；选择方案 A 时必须在 ADR 中显式记为“不适用”，不能伪造完成证据。

验收标准：

3. 不出现部分扣物、重复扣物、无依据 credit 或自动重试 `uncertain`。

### P0-05 修复撤离 saga 的并发与终态倒退 `[BE]`

当前 HTTP 结算和 Recovery Worker 可能同时处理同一条 `Consuming` 记录，而现有转换没有 expected state/revision CAS，终态可能被后到结果覆盖。

- [ ] 为 run 增加 `revision`、`leaseOwner`、`leaseExpiresAt`、`attempt` 和最后心跳。
- [ ] 原请求持有有效 lease 时，恢复任务不能接管；只允许回收过期 lease。
- [ ] 所有状态转换都必须携带 expected state 与 revision，使用 compare-and-swap。
- [ ] `Settled`、`Failed`、`Cancelled` 等终态不可降级回 `Removed`、`Uncertain` 或其他中间态。
- [ ] 钱包 credit、唯一账本记录和 `Removed -> Credited` 在同一数据库事务提交；`Credited -> Settled` 可安全恢复。
- [ ] 若账本已经存在该 run 的 credit，任何自动或人工路径都不得再标记 failed、退款或重复入账。
- [ ] 增加每玩家串行锁、全服有界消费队列和 backpressure。

验收标准：

1. 固定种子场景矩阵覆盖同 run 同键/异键、lease 过期接管、每个故障边界、多账户隔离和并发 1000 次；不出现重复命令、跨账户阻塞或终态倒退。
2. 不存在“账本已入账但 run 为 failed/uncertain”的组合。
3. 服务重启只能接管过期 lease，不能重新发送已经 dispatched 的游戏命令。

### P0-06 让商城发货具备可归因的结构化 receipt `[BE][PALDEFENDER/NATIVE]`

当前按背包总量 `current >= baseline + expectedDelta` 判断发货，玩家自然拾取相同物品可能误判成功；收到后立即使用又可能误进 `uncertain`。

- [ ] 选择 PalDefender 扩展或 Native grant adapter，返回绑定 `deliveryId` 和 `idempotencyKey` 的逐物品结构化 receipt。
- [ ] receipt 必须包含目标 PlayerUID、请求数量、实际 Granted 数量、命令版本和稳定结果 ID。
- [ ] 重放相同 delivery key 返回同一 receipt，同键不同请求拒绝。
- [ ] 部分成功进入人工对账，不能按“明确失败”自动全额退款。
- [ ] 发货回读超时从命令完成/ACK 时开始计算，不从基线采集时开始。
- [ ] 给发货队列增加容量、每玩家串行、dead-letter、最老任务年龄和熔断指标。

验收标准：

1. 自然拾取、移动、丢弃或立即消耗相同 SKU 都不影响发货归因。
2. 正常、明确失败、部分成功、响应丢失和超时路径都不会产生免费物品或重复扣款。
3. 连续 200 次测试发货无重复、无余额差异；每个故障点至少重复 20 次。

- [ ] 同一个门禁也约束后台发货 worker，避免版本漂移后“先扣款再失败”。

验收标准：

验收标准：
