# 05：实施阶段与验收

## 当前开发服实现证据（2026-07-15）

- 本机 SQLite 事务事件数据库、双货币账本、商品、限购、订单、发货状态与页面已接入；旧 JSONL 事件已验证可无损迁移。
- 同一商城幂等键重放返回同一订单；真实发货后 REST 背包增量已证明。
- Native MOD `0.3.0-dev.36` 已实现完整槽位快照、整单写前校验、普通静态资源槽位归零、回读与回滚；Control API 已持久化稳定请求键、request hash 和完整结果。
- Native 当前仅声明 `inventory.consume.experimental` 且 `persistenceVerified=false`。真实玩家的“扣物 → 保存 → 停服 → 重启 → 重登”未验收，所以生产资源兑换闸门仍必须关闭。
- 2026-07-12 曾在可信开发环境用 5 个 `Leather` 完成 RCON `/delitems` + REST 差值联调。该记录只证明早期玩法闭环，不是 Native 生产结算或持久化验收证据。
- 已实现玩家登录/session/CSRF/限流、账户隔离、角色权限/TOTP/审计、新人活动和双钱包事务；对外开放时仍需 HTTPS 反向代理与实际 Steam 登录运营配置。
- 已提供维护闸门、持久化 rollover 状态机、经济/世界一致性备份和可恢复执行脚本。仍需在真实开发世界完成 3 次换档与独立恢复演练。
- 当前存储适合单 Control API/worker 实例；只有需要多实例高可用时才将 SQLite 迁移到 PostgreSQL + transactional outbox + lease。

## 1. 推荐模块结构

实现时仍把玩法代码收拢在 `extraction-mode`，通过受限客户端调用现有 Pal Control，不把经济逻辑塞进当前单文件 API：

```text
extraction-mode/
├─ README.md
├─ docs/
├─ src/
│  ├─ Extraction.Api/             玩家和运营 HTTP API
│  ├─ Extraction.Domain/          账户、钱包、订单、资源兑换、赛季状态机
│  ├─ Extraction.Infrastructure/  SQLite、Pal Control client、Outbox
│  └─ Extraction.Worker/          发货、扣物、日刷新、换档编排
├─ web/                            玩家商城与资源兑换页面
├─ db/migrations/                  版本化存储迁移（多节点时增加 PostgreSQL）
├─ deploy/                         服务和反向代理示例，不含密钥
└─ tests/
   ├─ unit/
   ├─ contract/
   ├─ integration/
   ├─ e2e/
   └─ fault-injection/
```

MVP 的 Native 消费适配器和 Control API 端点放在现有 `mods/pal-control-native`、`services/control-api` 与共享契约中，因为它们属于游戏适配层；玩法模块只依赖固定 `IInventoryConsumePort` DTO，不接触 Unreal 指针、RCON 字符串或密码。RCON 兼容代码仅保留为 Development 诊断，不能通过生产资源兑换闸门。

## 2. 阶段 0：冻结边界与实验验证

目标：在写经济代码前证明关键上游能力，避免把未验证假设做成账本协议。

交付：

- 固定游戏 build、PalDefender、Native MOD、bridge 协议与编译依赖组合。
- 保存 PalDefender players/items/give-items 的脱敏 fixture。
- 定义 `IPlayerDirectory`、`IInventoryReader`、`IItemGrantPort`、`IInventoryConsumePort` 四个端口。
- 为 `inventory.consume` DTO 写 JSON Schema、错误码、dry-run 契约、完整槽位快照契约、稳定键和 capability/persistence 探针；MVP 生产后端固定为 Native。
- 记录 Named Pipe ACL、游戏/MOD/协议版本锁定、编译依赖 hash 与不向公网暴露 Control API/PalDefender/RCON 的安全基线。
- 确认 PalDefender MapLocation 坐标与地图资源兑换区坐标系一致。

验收：

1. 测试账户能通过同一响应获得完全匹配的 `UserId` 与 `PlayerUID`。
2. 六个容器的规范化快照对属性顺序不敏感，相同内容产生相同 SHA-256。
3. `give/items` 在测试账户上返回结构化 Granted 结果，现有命令查询能观察完整终态。
4. 位置在区内/区外的 10 组人工样本全部判定正确。
5. 受控玩家通过 Native 删除 1 个低价值普通静态物品，完整槽位前后快照精确证明差值。
6. 执行官方保存、停服、重启和玩家重登，证明被扣物品未恢复，才能将 hello 升级为稳定 capability 与 `persistenceVerified=true`。
7. 未批准版本、只有 experimental capability、快照不完整或 persistence 证据失效时，consume 必须 fail closed，不影响安全只读端点。

阶段 0 不执行永久钱包或正式商品交易。

## 3. 阶段 1：数据库、账本和认证

交付：

- SQLite 版本化迁移：账户、身份、赛季、钱包、账本、商品、订单、资源兑换、outbox、审计和调度表；多实例部署才增加 PostgreSQL 迁移。
- 钱包 repository 只暴露 credit/debit/reverse，不暴露 setBalance。
- Steam 登录、安全 Cookie、CSRF、账户隔离和本周角色绑定。
- 玩家门户与本机运营 API 物理/逻辑分离。
- 健康检查和购买/资源兑换独立闸门。

验收：

1. 100 个并发扣款请求竞争同一余额，成功总额不超过初始余额，最终余额与 ledger 求和一致。
2. 同 key 同请求并发 100 次只生成一条订单/账本；同 key 不同请求稳定返回 409。
3. 永久钱包不带 season，赛季钱包必须带 season，数据库约束能阻止错误数据。
4. Steam A 不能读取 Steam B 的钱包、订单或资源兑换。
5. 伪造昵称、PlayerUID 或其他平台 UserId 无法绑定。
6. CSRF、无 HTTPS（生产配置）、过期 session 和速率限制测试通过。
7. 数据库不可写时经济闸门自动关闭，没有内存降级路径。

## 4. 阶段 2：商城发货闭环

交付：

- 日 rotation 与 offer 只读 API。
- 创建订单的扣款事务与限购。
- Pal Control 客户端、确定性发货键、outbox worker。
- 订单状态页面和运营对账页面。
- 失败退款与人工补偿流程。

验收场景：

| 场景 | 必须结果 |
| --- | --- |
| 正常购买 | 一次扣款、一次 PalDefender 发货、订单 fulfilled |
| 重复点击/刷新 | 返回同一订单，无重复扣款/发货 |
| 余额不足 | 409，无订单副作用 |
| 超限购 | 409，无扣款 |
| 玩家离线 | 未派发前等待；超时后一次退款 |
| 上游明确 4xx | failed_refunded，原 debit 与 reversal 均保留 |
| 调用前 worker 崩溃 | 恢复后使用同一 delivery key 发一次 |
| 上游执行后响应丢失 | delivery_uncertain，不退款、不重发 |
| Control API 重启 | commandId/确定性 key 可恢复，不生成新副作用 |
| 玩家收到后立即消耗物品 | 明确 Granted ACK 仍判 fulfilled |

准入门槛：连续 200 次测试发货无重复、无余额差异；每个故障点至少重复 20 次。

## 5. 阶段 3：Native 完整槽位消费与持久化证明

这是公开资源经济服的结算信任边界。生产只能使用 Native 稳定 `inventory.consume`；RCON `/delitems` 早期闭环仅保留为 Development 诊断历史，不能作为入账证据。

交付：

- Native 完整槽位契约：容器/索引/ItemID/数量/静态元数据/revision，不允许客户端传入任意内存地址或函数。
- 在同一游戏 Tick 先校验整单，再扣物；支持安全清空普通静态资源槽位，动态、腐坏中或元数据不完整槽位在任何写入前整单拒绝。
- 完整 before/after 槽位回读、聚合 requested/actualConsumed 比较、失败回滚和精确腐坏标记。
- Control API 在发送前持久化报价快照、request hash 和确定性键，在重启后返回相同结果；已受理后不确定时禁止换键重发。
- hello 只有在真实重启验收后才声明稳定 capability/persistence，并与已批准游戏/MOD/协议版本组合一起作为硬闸门。

验收：

1. 单 ItemID 部分扣除、扣至 0、多 ItemID/多容器整单均成功，requested/actualConsumed 与完整槽位差值精确一致。
2. expected revision、PlayerUID/session、容器/索引、ItemID、数量或元数据任一不匹配时，在任何写入前整单拒绝。
3. 动态 GUID、腐坏中、元数据不可证明、负数/0、重复槽位、目录外物品和超长批次均被拒绝。
4. 玩家下线、世界变化或报价后背包变化时 fail closed，不执行 Native 写入。
5. Native 返回 success 但完整槽位后读/数量未匹配时进入 uncertain，不入账。
6. Native 已受理后连接断开进入 uncertain；服务重启 20 次都只查询原键，不换键重扣。
7. 故障注入让部分写入或回滚无法证明时进入 manual_review，不按比例自动入账。
8. stable capability、persistence 证据、Named Pipe ACL、完整快照探针或版本锁任一失效时只关闭 consume，不影响安全只读接口。
9. 受控压力下 Native 游戏 Tick 有界队列返回 backpressure，不绕过队列直接写内存。
10. 真实玩家完成扣物后，依次官方保存、停服、重启、重登，物品依然不存在；脱敏验收证据与对应 build/MOD hash 一起归档。

## 6. 阶段 4：资源兑换结算闭环

交付：

- 区域配置与连续位置采样。
- 报价、价格快照、30 秒过期和背包 hash。
- consume saga、扣物后唯一入账、玩家锁和对账页面。
- 物品回收目录、买卖价防套利规则。

验收场景：

| 场景 | 必须结果 |
| --- | --- |
| 正常资源兑换 | 全部目标物品消失后，一次增加战备券 |
| 区外请求 | 409，无报价副作用 |
| 两次采样间离区 | 409，无报价 |
| 报价后移动/消耗/拾取 | commit 412，无扣物、无入账 |
| 重复 quote/commit | 同 key 返回同资源；最多一次 consume/credit |
| 扣物前崩溃 | 恢复使用同键，不重复 |
| 扣物后、入账前崩溃 | 恢复后只入账一次 |
| 扣物响应丢失 | consume_uncertain，不入账、不重扣 |
| 部分扣物（故障注入） | manual_review，不按比例自动入账 |
| 目录外/不可售物品 | 不报价、不扣除 |
| 商城物品再出售 | 回收价值不高于配置上限，不存在正套利环 |

必须做经济守恒测试：对每次已完成资源兑换，ledger credit 精确等于冻结 line 的总和；所有 credit 都有已证明的 consume 证据。

## 7. 阶段 5：日刷新与周换档

交付：

- 确定性日轮换生成器、草稿 diff、原子发布和 grace window。
- rollover 持久状态机、预检、闸门、保存/备份、受保护启停和世界身份绑定。
- 战备券 expiry、周奖励和旧世界归档索引。
- rollover 状态页、阶段重试、开放前回滚。

验收：

1. 同一业务日任务执行 20 次仍只有一个 rotation，内容 hash 相同。
2. 发布中崩溃时当前 pointer 只能指旧版或完整新版，不出现半套 offer。
3. 使用完整开发世界完成 3 次“旧档 -> 新档”演练，每次新 worldguid 都不同、旧目录和备份均完整。
4. 在每个 rollover 阶段注入崩溃，恢复后不跳步骤、不重复 expiry/奖励、不误删世界。
5. 存在任何 uncertain 或活动资源兑换时换档被硬阻止。
6. 新世界身份、版本、Native 稳定 capability/persistence 或完整快照探针任一失败时不得打开赛季。
7. `season_opened` 前可恢复旧世界；产生新交易后系统拒绝自动回滚。
8. 数据库备份和游戏备份各完成一次独立恢复演练，并对账备份时间点后的交易。

## 8. 阶段 6：封闭测试与上线

封闭测试至少覆盖一个完整周档，先 5–10 人，再扩大；不跳过周换档。

上线门槛：

- 所有阶段验收证据归档并由不同于实现者的人员复核。
- 关键 contract、integration、fault-injection、security 测试在固定版本组合上通过。
- 最近 7 天无未解释的钱包差异、重复发货、无依据入账或部分扣物。
- uncertain 对账演练完成，值班人员能按手册处理。
- 数据库和世界备份恢复演练在目标机器完成。
- 公开玩家入口经过 HTTPS、认证、CSRF、权限隔离、限流和日志脱敏检查。
- 当前 Control API、PalDefender 和官方 REST 均未直接公网监听；可选 RCON 诊断端口不参与正式兑换，防火墙必须阻断所有远程入站且无公网映射。
- 有一键关闭购买、关闭资源兑换的熔断操作，且不依赖重启服务器。

## 9. 测试分层

| 层 | 覆盖 |
| --- | --- |
| Unit | 价格、区域计算、规范哈希、状态转换、请求哈希、余额规则 |
| Database integration | 唯一约束、并发钱包、ledger、outbox 租约、迁移升级/回退 |
| Contract | 玩家 API、Pal Control client、PalDefender fixture、Native consume DTO/完整槽位/capability 契约 |
| Game integration | 固定服务端上的真实发货、Native 扣物、保存与重启后持久化 |
| E2E | 登录 -> 绑定 -> 购买 -> 发货 -> 资源兑换 -> 钱包到账 |

测试断言不能只检查 HTTP 状态。每个写场景同时检查：领域状态、wallet、ledger、outbox、游戏命令数、游戏实际物品、审计事件和重启后结果。

## 10. 性能与容量基线

MVP 目标不是高并发交易所，但需要明确上限：

- 创建订单数据库响应 p95 < 500 ms（不含游戏发货完成）。
- 正常发货终态 p95 < 5 s。
- 资源兑换报价含两次位置采样，p95 < 5 s。
- 正常资源兑换 commit 到到账 p95 < 5 s。
- 单账户游戏写入串行；Native consume 由游戏 Tick 有界队列限流，并按实测 Tick 延迟、队列深度和完整回读成本设定容量上限。发货继续由 Pal Control 队列限流。
- Outbox 正常最老消息 < 10 s，超过 60 s 自动告警和关闸。
- 单次订单最多 10 行；单次资源兑换最多 200 个目标槽位，超过时拒绝并提示分批整理。

## 11. 首发内容建议

为了先验证循环而非复杂平衡，首发只上：

- 10–20 个基础补给商品；
- 50–100 个有明确 ItemID 的可售资源；
- 2 个资源兑换区；
- 单一日轮换规则；
- 每账户小额初始商域币；
- 回收价上限为商店最低买价的 50%，高风险/目录外物品不可售。

前两个周档只调价和限购，不增加帕鲁或装备资源兑换。等账本、扣物和换档连续稳定后再扩展内容。
