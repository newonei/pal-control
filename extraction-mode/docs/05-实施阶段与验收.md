# 05：实施阶段与验收

## 当前开发服实现证据（2026-07-12）

- 本机 SQLite 事务事件数据库、双货币账本、商品、限购、订单、发货状态与页面已接入；旧 JSONL 事件已验证可无损迁移。
- 同一商城幂等键重放返回同一订单；真实发货后 REST 背包增量已证明。
- RCON 仅本机适配器已通过协议探针；Windows 防火墙阻断非 loopback 的 `25575/TCP`。
- 真实使用 5 个 `Leather` 完成“区域双采样 → 30 秒报价 → `delitems` → REST 数量归零 → 战备券 +10”；重复结算未重复入账。
- 已提供维护闸门、换档 readiness/commit API 和 `scripts/Invoke-WeeklyRollover.ps1`。尚未在本轮实际执行换新世界，避免无必要重置当前开发档。
- 当前存储适合单机开发；公网玩家门户、Steam 登录/RBAC、Native 原子消费、PostgreSQL、多实例高可用与完整故障注入仍属于正式上线前工作。RCON `delitems` + REST 差值只作为可信管理员环境下的玩法联调通道。

## 1. 推荐模块结构

实现时仍把玩法代码收拢在 `extraction-mode`，通过受限客户端调用现有 Pal Control，不把经济逻辑塞进当前单文件 API：

```text
extraction-mode/
├─ README.md
├─ docs/
├─ src/
│  ├─ Extraction.Api/             玩家和运营 HTTP API
│  ├─ Extraction.Domain/          账户、钱包、订单、撤离、赛季状态机
│  ├─ Extraction.Infrastructure/  PostgreSQL、Pal Control client、Outbox
│  └─ Extraction.Worker/          发货、扣物、日刷新、换档编排
├─ web/                            玩家商城与撤离页面
├─ db/migrations/                  版本化 PostgreSQL 迁移
├─ deploy/                         服务和反向代理示例，不含密钥
└─ tests/
   ├─ unit/
   ├─ contract/
   ├─ integration/
   ├─ e2e/
   └─ fault-injection/
```

MVP 的白名单 RCON 消费适配器和对应 Control API 端点放在现有 `services/control-api` 与共享契约中，因为它们属于游戏适配层；玩法模块只依赖固定 `IInventoryConsumePort` DTO，不接触 RCON 字符串或密码。后续 Native `inventory.consume` 可在 `mods/pal-control-native` 实现为第二种后端，不改变玩法契约。

## 2. 阶段 0：冻结边界与实验验证

目标：在写经济代码前证明关键上游能力，避免把未验证假设做成账本协议。

交付：

- 固定游戏 build、PalDefender、RCON 适配器版本；启用 Native 能力时另固定 MOD 和 bridge 协议组合。
- 保存 PalDefender players/items/give-items 的脱敏 fixture。
- 定义 `IPlayerDirectory`、`IInventoryReader`、`IItemGrantPort`、`IInventoryConsumePort` 四个端口。
- 为通用 `inventory.consume` DTO 写 JSON Schema、错误码、dry-run 契约和探针；MVP 后端固定为 `paldefender-rcon-delitems`。
- 记录 RCON 本机连接、防火墙远程阻断、无端口映射和 AdminPassword Secret Store 的安全基线。
- 确认 PalDefender MapLocation 坐标与地图撤离区坐标系一致。

验收：

1. 测试账户能通过同一响应获得完全匹配的 `UserId` 与 `PlayerUID`。
2. 六个容器的规范化快照对属性顺序不敏感，相同内容产生相同 SHA-256。
3. `give/items` 在测试账户上返回结构化 Granted 结果，现有命令查询能观察完整终态。
4. 位置在区内/区外的 10 组人工样本全部判定正确。
5. 测试账户通过 `/delitems <UserId> ItemId:Amount` 删除 1 个低价值物品，REST 前后快照精确证明差值。
6. 从另一主机连接 RCON 失败，本机适配器成功；扫描确认没有公网端口映射。
7. 未批准版本或 RCON 安全检查失效时四个端口全部 fail closed。

阶段 0 不执行永久钱包或正式商品交易。

## 3. 阶段 1：数据库、账本和认证

交付：

- PostgreSQL 迁移：账户、身份、赛季、钱包、账本、商品、订单、撤离、outbox、审计和调度表。
- 钱包 repository 只暴露 credit/debit/reverse，不暴露 setBalance。
- Steam 登录、安全 Cookie、CSRF、账户隔离和本周角色绑定。
- 玩家门户与本机运营 API 物理/逻辑分离。
- 健康检查和购买/撤离独立闸门。

验收：

1. 100 个并发扣款请求竞争同一余额，成功总额不超过初始余额，最终余额与 ledger 求和一致。
2. 同 key 同请求并发 100 次只生成一条订单/账本；同 key 不同请求稳定返回 409。
3. 永久钱包不带 season，赛季钱包必须带 season，数据库约束能阻止错误数据。
4. Steam A 不能读取 Steam B 的钱包、订单或撤离。
5. 伪造昵称、PlayerUID 或其他平台 UserId 无法绑定。
6. CSRF、无 HTTPS（生产配置）、过期 session 和速率限制测试通过。
7. 数据库不可写时经济闸门自动关闭，没有内存降级路径。

## 4. 阶段 2：商城发货闭环

交付：

- 日 rotation 与 offer 只读 API。
- 创建订单的扣款事务与限购。
- Pal Control 客户端、确定性发货键、outbox worker。
- 订单状态页面和运营对账页面。
- 失败退款与人工补偿流程。

验收场景：

| 场景 | 必须结果 |
| --- | --- |
| 正常购买 | 一次扣款、一次 PalDefender 发货、订单 fulfilled |
| 重复点击/刷新 | 返回同一订单，无重复扣款/发货 |
| 余额不足 | 409，无订单副作用 |
| 超限购 | 409，无扣款 |
| 玩家离线 | 未派发前等待；超时后一次退款 |
| 上游明确 4xx | failed_refunded，原 debit 与 reversal 均保留 |
| 调用前 worker 崩溃 | 恢复后使用同一 delivery key 发一次 |
| 上游执行后响应丢失 | delivery_uncertain，不退款、不重发 |
| Control API 重启 | commandId/确定性 key 可恢复，不生成新副作用 |
| 玩家收到后立即消耗物品 | 明确 Granted ACK 仍判 fulfilled |

准入门槛：连续 200 次测试发货无重复、无余额差异；每个故障点至少重复 20 次。

## 5. 阶段 3：本机 RCON 扣物与 REST 证明

这是本机开发服闭环的集成通道，不是公开经济服的最终信任边界。正式上线必须用 Native 原子消费或服务端托管撤离容器替换 RCON 归因。

交付：

- Palworld `AdminPassword` 的强凭据与 Secret Store、本机客户端、防火墙远程阻断、无公网映射和可失效的安全检查。
- Control API 结构化消费端点，只接受 UserId、expectedSnapshotHash 和白名单 ItemID/Amount；不接受任意命令文本。
- 命令构造器只输出一条 `/delitems <UserId> ItemId:Amount...`，并限制条目、数量、字符集和总长度。
- `/clearinv` 不在自动 API 白名单；只在开发服/人工应急流程保留。
- PalDefender REST before/after 全六容器快照与规范 hash。
- 写入 socket 前持久化、写入后 `uncertain` 禁止重发、部分删除人工对账。
- 后端 capability、只读 dry-run、安全状态和版本 probe。

验收：

1. 单 ItemID 部分扣除、扣至 0、多 ItemID 单命令均成功，并由 REST 总量差值精确证明。
2. expectedSnapshotHash、UserId、ItemID 或可用数量任一不匹配时，在发送前拒绝。
3. 昵称目标、负数、0、重复 ItemID、目录外 ItemID、命令分隔符、换行、超长批次和 `/clearinv` 均被拒绝。
4. 玩家下线、世界变化或报价后背包变化时 fail closed，不发送 RCON。
5. RCON 文本称成功但 REST 后读未变化时进入 uncertain，不入账。
6. RCON 开始写入后连接断开进入 uncertain；服务重启 20 次都不再次发送。
7. 故障注入让 `/delitems` 只完成部分条目时进入 manual_review，不按比例自动入账。
8. 防火墙远程允许、检测到端口映射、Secret 缺失或版本漂移时只关闭 consume，不影响只读接口。
9. 受控压力下 RCON 有界队列保持全局串行并返回 backpressure，不交错后快照。

RCON 没有跨 ItemID 原子事务，也没有结构化幂等 ACK；本机版本实测只返回 `Success.`，不返回实际删除数量。严格白名单、单命令和 REST 后快照能发现失败与部分差值，但无法证明数量减少一定由该命令造成：恶意玩家可能在前快照和命令之间丢弃或转移物品。因此它只允许开发服联调，任何中断或部分差值均冻结人工对账，公开经济测试必须切换 Native 原子消费。

可选阶段 3B 再开发 Native `inventory.consume`：同一游戏 Tick 内先验证全部槽位，再允许归零、结算和逐槽位回读。替换后仍必须保留 after snapshot、`uncertain` 和故障注入测试。

## 6. 阶段 4：撤离结算闭环

交付：

- 区域配置与连续位置采样。
- 报价、价格快照、30 秒过期和背包 hash。
- consume saga、扣物后唯一入账、玩家锁和对账页面。
- 物品回收目录、买卖价防套利规则。

验收场景：

| 场景 | 必须结果 |
| --- | --- |
| 正常撤离 | 全部目标物品消失后，一次增加战备券 |
| 区外请求 | 409，无报价副作用 |
| 两次采样间离区 | 409，无报价 |
| 报价后移动/消耗/拾取 | commit 412，无扣物、无入账 |
| 重复 quote/commit | 同 key 返回同资源；最多一次 consume/credit |
| 扣物前崩溃 | 恢复使用同键，不重复 |
| 扣物后、入账前崩溃 | 恢复后只入账一次 |
| 扣物响应丢失 | consume_uncertain，不入账、不重扣 |
| 部分扣物（故障注入） | manual_review，不按比例自动入账 |
| 目录外/不可售物品 | 不报价、不扣除 |
| 商城物品再出售 | 回收价值不高于配置上限，不存在正套利环 |

必须做经济守恒测试：对每次已完成撤离，ledger credit 精确等于冻结 line 的总和；所有 credit 都有已证明的 consume 证据。

## 7. 阶段 5：日刷新与周换档

交付：

- 确定性日轮换生成器、草稿 diff、原子发布和 grace window。
- rollover 持久状态机、预检、闸门、保存/备份、受保护启停和世界身份绑定。
- 战备券 expiry、周奖励和旧世界归档索引。
- rollover 状态页、阶段重试、开放前回滚。

验收：

1. 同一业务日任务执行 20 次仍只有一个 rotation，内容 hash 相同。
2. 发布中崩溃时当前 pointer 只能指旧版或完整新版，不出现半套 offer。
3. 使用完整开发世界完成 3 次“旧档 -> 新档”演练，每次新 worldguid 都不同、旧目录和备份均完整。
4. 在每个 rollover 阶段注入崩溃，恢复后不跳步骤、不重复 expiry/奖励、不误删世界。
5. 存在任何 uncertain 或活动撤离时换档被硬阻止。
6. 新世界身份、版本、RCON 本机隔离或 consume/REST 回读 probe 任一失败时不得打开赛季。
7. `season_opened` 前可恢复旧世界；产生新交易后系统拒绝自动回滚。
8. 数据库备份和游戏备份各完成一次独立恢复演练，并对账备份时间点后的交易。

## 8. 阶段 6：封闭测试与上线

封闭测试至少覆盖一个完整周档，先 5–10 人，再扩大；不跳过周换档。

上线门槛：

- 所有阶段验收证据归档并由不同于实现者的人员复核。
- 关键 contract、integration、fault-injection、security 测试在固定版本组合上通过。
- 最近 7 天无未解释的钱包差异、重复发货、无依据入账或部分扣物。
- uncertain 对账演练完成，值班人员能按手册处理。
- 数据库和世界备份恢复演练在目标机器完成。
- 公开玩家入口经过 HTTPS、认证、CSRF、权限隔离、限流和日志脱敏检查。
- 当前 Control API、PalDefender 和官方 REST 均未直接公网监听；RCON 只能由本机适配器连接，防火墙阻断远程且无端口映射。
- 有一键关闭购买、关闭撤离的熔断操作，且不依赖重启服务器。

## 9. 测试分层

| 层 | 覆盖 |
| --- | --- |
| Unit | 价格、区域计算、规范哈希、状态转换、请求哈希、余额规则 |
| Database integration | 唯一约束、并发钱包、ledger、outbox 租约、迁移升级/回退 |
| Contract | 玩家 API、Pal Control client、PalDefender fixture、RCON consume DTO/命令白名单 |
| Game integration | 固定服务端上的真实发货、RCON `/delitems`、REST 前后背包 |
| E2E | 登录 -> 绑定 -> 购买 -> 发货 -> 撤离 -> 钱包到账 |
| Fault injection | 每个持久化/派发/ACK/入账边界崩溃、断网和超时 |
| Security | 越权、CSRF、重放、IDOR、限流、秘密泄漏、路径/命令注入 |
| Rollover rehearsal | 完整旧世界归档、新世界启动、验证、开放和回滚边界 |

测试断言不能只检查 HTTP 状态。每个写场景同时检查：领域状态、wallet、ledger、outbox、游戏命令数、游戏实际物品、审计事件和重启后结果。

## 10. 性能与容量基线

MVP 目标不是高并发交易所，但需要明确上限：

- 创建订单数据库响应 p95 < 500 ms（不含游戏发货完成）。
- 正常发货终态 p95 < 5 s。
- 撤离报价含两次位置采样，p95 < 5 s。
- 正常撤离 commit 到到账 p95 < 5 s。
- 单账户游戏写入串行；MVP 的 RCON consume 全服串行，初始建议不超过 1 command/s，经 REST 后读压力测试再调整。发货继续由现有 Pal Control 队列限流。
- Outbox 正常最老消息 < 10 s，超过 60 s 自动告警和关闸。
- 单次订单最多 10 行；单次撤离最多 200 个目标槽位，超过时拒绝并提示分批整理。

## 11. 首发内容建议

为了先验证循环而非复杂平衡，首发只上：

- 10–20 个基础补给商品；
- 50–100 个有明确 ItemID 的可售资源；
- 2 个撤离区；
- 单一日轮换规则；
- 每账户小额初始商域币；
- 回收价上限为商店最低买价的 50%，高风险/目录外物品不可售。

前两个周档只调价和限购，不增加帕鲁或装备撤离。等账本、扣物和换档连续稳定后再扩展内容。
