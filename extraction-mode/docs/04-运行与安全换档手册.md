# 04：运行与安全换档手册

> 商城购买与资源兑换的统一运行时门禁、稳定 blocker code 和独立熔断操作见 [Economy Safety Gate 运行手册](../../docs/runbooks/economy-safety-gate.md)。内容草稿、维护排空、发布失败重放和回滚见 [经济内容发布与回滚运行手册](../../docs/runbooks/economy-content-publishing.md)。换档维护是全局写闸门；`purchase` 与 `resource-exchange` 熔断器用于不重启服务地隔离单一玩法写路径。

## 1. 时间与职责

- 业务时区固定 `Asia/Shanghai`，数据库时间保存 UTC。
- 日刷新：每天 `04:00`。
- 周维护开始：周一 `03:30`；目标新档开放：`04:00`。
- 值班角色至少分为 Operator（观察/关闭闸门）、EconomyAdmin（对账/钱包补偿）、SeasonAdmin（换档）。生产环境中钱包调整和周档开放建议双人审批。

计划任务只负责“触发一个有唯一 schedule key 的运行记录”，不得直接执行一串无状态脚本。进程重启后从 `scheduled_runs` 和 rollover 状态继续。

## 2. 每日例行检查

启用 `ExtractionMode` 的每次 Control API 启动都会在 HTTP 流量进入前确保 current content 已初始化。全新开源安装必须先生成经授权的 `services/control-api/Resources/palworld-resource-catalog.json`（确认外部数据条款后运行 `.\tools\catalogs\update-palworld-resource-catalog.ps1`，或提供替代快照）；目录缺失/非法或首版原子激活失败时，服务必须启动失败，不能以空商城继续运行。

### 2.1 开服健康

检查现有本机接口：

```powershell
$Api = 'http://127.0.0.1:5180/api/v1'
$AdminApiKey = Read-Host "Control API 只读管理密钥" -AsSecureString
$AdminCredential = [pscredential]::new("viewer", $AdminApiKey)
$AdminHeaders = @{
  "X-Pal-Admin-Key" = $AdminCredential.GetNetworkCredential().Password
}
Invoke-RestMethod 'http://127.0.0.1:5180/health/ready'
Invoke-RestMethod "$Api/servers/local/info" -Headers $AdminHeaders
Invoke-RestMethod "$Api/servers/local/paldefender/status" -Headers $AdminHeaders
Invoke-RestMethod "$Api/servers/local/bridge/status" -Headers $AdminHeaders
Invoke-RestMethod "$Api/servers/local/saves/status" -Headers $AdminHeaders
Invoke-RestMethod "$Api/servers/local/economy-content/current" -Headers $AdminHeaders
$Settlement = Invoke-RestMethod "$Api/extraction/admin/settlement/status" -Headers $AdminHeaders
$Settlement | Select-Object adapter, enabled, connected, outcome, error
```

密钥应从密码管理器粘贴到 `SecureString` 提示，关闭巡检终端后即丢弃；不要把它写入脚本、命令参数、仓库或截图。高风险写操作还必须按对应手册单独提供 TOTP 与审计原因。

Production 必须显示 `adapter=native`；只有同时满足宿主为 Development、`Security:DevelopmentMode=true`、`PlayerPortal:PublicSteam=false`、RCON 已启用、`ExtractionMode:Rcon:AllowDevelopmentSettlement=true` 且未强制 Native 的隔离诊断环境才允许显示 `adapter=development-rcon`。`connected=true` 只表示当前探针成功，不替代稳定 `inventory.consume`、`persistenceVerified=true`、批准版本组合和真实保存/重启验收。旧 `/extraction/admin/rcon/status` 是已废弃的同结构别名，只供旧脚本迁移，新的巡检与控制台不得使用。

值班人还需在玩法状态页确认：

- 数据库可写、schema 版本正确；
- 当前开放赛季 `world_guid` 等于官方 REST；
- 当前游戏、商城发货所用 PalDefender，以及资源兑换所用 Native 协议/game build/MOD 版本等于已批准组合；
- Native 只通过本机 Named Pipe 访问，并声明 `inventory.probe` 与经过真实持久化验收的稳定 `inventory.consume`；可选 RCON 诊断端口仍由防火墙阻断远程入站且没有公网映射；
- 两个经济闸门状态及关闭原因；
- outbox 最老消息、近 15 分钟失败率、uncertain 数量；
- 本日 rotation 已发布且内容 hash 不为空；
- 最近一次数据库备份与游戏受管备份可用。

### 2.2 日刷新

当前自动日刷新不是 `03:55` 预生成脚本：第一次在 `04:00` 业务日边界后访问内容相关路径时，运行时会从上一不可变定义创建新营业日草稿；准备请求键包含新草稿 ID，SQLite 以 `(server_id, business_date, content_hash)` 唯一约束和 current pointer CAS 保证同一营业日内容不会重复激活，再在一个事务中提交 pointer 与整套商品投影。商品有效价格由 `businessDate + SKU + rulesVersion` 的确定性 SHA-256 结果选择 90%、95%、100%、105% 或 110%；相同输入永远得到相同价格。日任务实例使用营业日周期键，周任务使用周档周期键。

值班人应在边界前完成：

1. 读取当前版本、完整 content hash、依赖版本和 diff；确认授权目录、价格、限购、全服库存、兑换区窗口/倍率和任务仍有效。
2. 运行严格校验。直接回售检查会按数量、结算取整和最高区域倍率阻断可证明正收益；它不覆盖制作、加工、拆包、跨币种或玩家交易。
3. 确认双候选点中至少一个当前可用，并核对今日热点、有效收益倍率、路线和风险提示。自动化已覆盖双点相邻业务日轮换、全关 next-open 与 closing/grace 边界；候选点 2 的真实坐标、路线和半径仍须按 P1-01 实服验收。

边界后读取 current pointer、玩家目录与任务，核对营业日、规则版本、content hash 和版本 ID 一致。旧页面提交旧商品 offer 时服务端以 `OFFER_NOT_AVAILABLE` 拒绝；既有订单继续使用冻结行，不会被新价格改写。

内容激活已有第 N 个商品故障注入、重启重试、20 次激活与回滚自动化：商品投影事件、current pointer 和激活记录同事务提交，失败时仍对外展示完整旧版，不会出现半套 offer；系统仍没有 6 小时自动 grace 承诺。日刷新或人工激活出现错误时立即关闭购买与资源兑换并保持维护：先读取 current pointer 与目录证据，确认仍为整套旧版后，用原幂等请求重试同一已准备版本；成功后再次只读核对，内容本身有误时再回滚到已有完整版本。核对完成前不得开放交易。

### 2.3 选择性出售巡检与响应丢失

- 玩家先取得包含全部合格资源的源报价，再以 `sourceRevision + items[]` 派生所选子报价。源报价取消与子报价创建必须同时可见；发现源仍为 `Quoted` 但已有子报价、同一源多个子报价，或半提交证据时立即关闭资源兑换。
- 玩家报告“确认后页面报错”时，先用原 `Idempotency-Key` 查询/重放，不能生成新键重新选择。完全相同请求应跨重启返回同一 `runId`；同键换账户、源报价或选择内容必须返回冲突。
- Native 证据中 `expectedContainers` 应保留扫描时三个容器的完整快照/hash，而 `items` 必须只含玩家所选 ItemID/数量。Development RCON 只能在既有隔离条件下发送所选行，不能据完整快照删除未选物品。
- 选择与结算并发时只能有一个胜者：源进入 `Cancelled` 并产生一个子报价，或源直接进入 `Consuming` 且没有子报价。异常状态按 [选择性资源出售运行手册](../../docs/runbooks/selective-resource-sale.md) 关闭闸门并对账。

## 3. Native 扣物验收与可选 RCON 隔离

正式资源兑换只使用通过门禁的稳定 Native `inventory.consume`。旧 dev36 曾处理完整快照和普通静态槽位清空，但仅为 experimental 且 `persistenceVerified=false`；dev37-ro 因离线库存边界缺陷已淘汰。dev38-ro 曾完成 9 项非玩家操作和 3 项无人在线拒绝，但现已 superseded/quarantined。当前 dev39-ro 更严格地只声明只读探针，不发布写能力；它只有双独立可复现制品证据，固定运行套件、在线玩家三项、PalDefender 组合和独立复核仍缺失。只读门禁、独立写候选和真实持久化验收全部完成前，资源兑换写闸门保持关闭；官方保存与优雅关服只证明历史维护流程。

1. 记录精确 Palworld、UE4SS、Unreal、MOD、Bridge 协议与 DLL SHA-256；任何变化使验收失效。
2. 只读验证 hello/heartbeat、PlayerUID 唯一映射、三个必需容器、所有槽位动态元数据和规范快照 hash。
3. 使用专用在线测试账户和低价值白名单物品执行一次扣物，保存逐行、逐槽位、完整快照和聚合回执。
4. 请求官方保存，正常停服、重启并重新登录；确认物品没有恢复，且相同 Native/Control API 幂等键只返回原结果、不再次扣物。
5. 只有上述证据通过，才在 Native hello 中声明稳定 `inventory.consume`、返回 `persistenceVerified=true`，并同步生产批准版本。不能只改配置伪造能力。
6. transport 中断、回滚未证明、动态/腐化槽位、回读或持久化证据不全必须进入 `Uncertain`，不能自动重发或入账。

RCON 只保留给 Development 诊断和经批准的人工应急：Palworld 的 RCON 密码来自 `AdminPassword`，只能由 Control API 通过 loopback 连接；Windows 防火墙、路由器和云安全组不得开放其 TCP 端口。Production 不会在 Native 不可用时 fallback 到 `/delitems`，玩家门户和浏览器也永远不能持有 RCON 密码或提交原始命令。

值班判断资源结算健康时只读取 `/api/v1/extraction/admin/settlement/status` 的通用状态；无论底层是 Native 还是 Development RCON，错误输出都不得包含连接 Secret、原始命令或内部传输地址。

## 4. 不确定交易处置

### 4.1 发货 uncertain

1. 关闭该账户的新购买；不要全服重启或再次点击发货。
2. 保存 order、delivery、Pal Control commandId、确定性幂等键、请求 payload 和时间窗口。
3. 查询原命令状态与 append-only 事件；检查 PalDefender 同时间段日志。
4. 读取当前背包仅作为辅助，不把“现在没有物品”当成未发货证明。
5. 证据明确成功：对账为 `fulfilled`。
6. 证据明确上游从未执行：产生 `shop_reversal` 并转 `failed_refunded`。
7. 证据不足：转 `manual_review`，由两名授权人员决定是否创建独立补偿发货。
8. 记录 reason、证据摘要和审批者后解锁账户。

当前本机开发 API 的终结操作必须先开启 maintenance，并精确填写确认串：

- 已证明发货成功：`POST /api/v1/extraction/admin/orders/{orderId}/reconcile`，`resolution=delivered`，`confirmation=ORDER-{orderId无连字符小写}-DELIVERED`。
- 已证明没有发货：同端点使用 `resolution=refund`、固定 `Idempotency-Key` 与 `confirmation=ORDER-{orderId无连字符小写}-REFUND`；退款与订单终态在同一个 SQLite 事件中提交。

这些接口使用管理身份、RBAC、TOTP、确认短语和持久审计，但当前还没有双人审批工作流；正式服高价值人工入账应在双人审批完成前保持关闭。

禁止：删除旧订单、把 uncertain 直接改成 failed、复用旧 payload 但生成新 key 自动重发。

### 4.2 扣物 uncertain

只要 Native `inventory.consume` 已经派发，随后出现超时、断线、进程重启、回执持久化失败或任何证据无法验证，都按 `Uncertain` 处理；不能因为客户端报错就推断游戏没有执行。

1. 关闭该账户购买与资源兑换；不要让玩家移动目标物品，发送定向提示。
2. 保存完整 Native 报价快照、request hash、稳定 result ID、派发阶段、逐行/槽位回执和所有目标 ItemID/数量。
3. 查询已持久化的原回执和 run 状态，不生成第二个 consume key，也不再次调用 Native。
4. 由管理员结合重新登录后的三个容器快照、官方保存/重启证据和账本状态人工复核；自然移动或消耗不能单独作为归因证据。
5. 只有原 Native 成功回执已在扣物后持久化且全部证据一致，恢复 worker 才继续唯一 `extraction_credit`；否则保持人工对账。
6. 全部未扣且明确未派发：取消，不入账。
7. 部分扣除或物品已移动：保持 `Uncertain`。选择“补齐扣除后入账”或“补发已扣部分并取消”前必须双人复核，所有补偿均为新记录。
8. 完成审计后解锁账户。

当前本机开发 API 可在 maintenance 下调用 `POST /api/v1/extraction/admin/runs/{runId}/reconcile`：已证明未扣物用 `resolution=failed` 和 `RUN-{runId无连字符小写}-FAILED`；已证明全部扣物用 `resolution=settled` 和 `RUN-{runId无连字符小写}-SETTLED`。后者以 runId 固定幂等键补发战备券，崩溃重试不会重复入账。证据不足时不得调用，继续保持 `Uncertain`。

## 5. 周档安全切换

### 5.1 阶段与硬闸门

rollover 状态：

```text
planned -> preflight_passed -> economy_closed -> old_world_saved
        -> old_world_archived -> old_server_stopped -> new_world_started
        -> new_world_verified -> season_opened -> completed
```

任一阶段失败进入 `failed_at_<stage>`，保留所有证据。每一步都有唯一 step key，重复执行只能返回原结果或继续未完成步骤。

以下条件不满足时不得开始：

- 没有未完成/不确定发货，也没有处于 `Consuming/Uncertain/Removed/Credited` 的资源 run；
- outbox 已清空且无过期租约；
- 数据库全量备份完成并做过恢复验证；
- 当前世界受管备份功能健康、磁盘空间满足保留策略；
- 新旧赛季时间、配置 hash、版本白名单和新 DedicatedServerName 已批准；
- Native 精确版本、稳定能力和最近一次真实持久化验收仍有效；可选 RCON 诊断端口继续满足远程阻断、无映射和 Secret 检查；
- 有足够维护窗口和当班 SeasonAdmin。

### 5.2 T-30 分钟：关闭经济

1. 广播维护倒计时。
2. 把旧赛季改为 `closing`；关闭购买、报价和 commit。
3. 等待尚未派发的订单自然终结，最多 10 分钟。
4. 重新检查 uncertain、active extraction、outbox 和玩家绑定写入为 0。
5. 为剩余战备券生成 `expiry` 分录清零；先完成已批准周奖励的商域币分录。
6. 生成旧赛季结算摘要和 hash，不再允许修改统计。

若第 4 步不通过，**推迟换档**，不要带着未知游戏副作用切到新世界。

### 5.3 保存与归档旧世界

通过现有存档中心请求“保存并创建备份”，等待终态 `completed`。只有 manifest 和逐文件 SHA-256 校验通过才可继续。

需要保存的证据：

- 旧赛季 ID、旧 `world_guid`、DedicatedServerName；
- 官方 REST info 和运行进程路径；
- 受管备份 ID、manifest 路径、文件数、总大小与根 hash；
- 游戏、PalDefender、Native 协议/game build/MOD、DLL SHA-256 与服务器配置版本；
- 数据库备份 ID 和校验结果；
- 结算摘要 hash。

旧世界目录不能删除、移动或覆盖。受管备份必须位于活动 SaveGames 目录之外。归档保留期首版建议至少 8 周。

### 5.4 正常停止旧服务器

1. 用官方 REST 正常关服，包含维护说明和合理倒计时。
2. 等待 PalServer 进程退出；确认游戏、Query、RCON 和本机管理端口均已释放。
3. 不以强杀作为正常路径。超时进入失败状态，值班人先保留日志和进程转储，再依据服务器运行手册决定是否终止。
4. 记录实际退出时间和退出码。

### 5.5 准备新世界

1. 为新赛季生成全新的、符合 Palworld 约束的 DedicatedServerName；不得复用旧值。
2. 在 staging 文件生成新配置，校验语法、允许的游戏设置、端口和目录边界；计算 config hash。
3. 只修改选择新世界所必需的配置。旧活动世界保持原位，不能通过删除旧目录“迫使开新档”。
4. 用受保护启动脚本启动固定版本组合。
5. 等待官方 REST、商城所需 PalDefender REST 和 Native Bridge 全部 ready；若配置了 RCON 诊断则只做隔离检查。启动超时则停止新进程并保留日志。

具体配置写入应由新增 rollover adapter 实现：先创建临时文件、刷新磁盘、原子替换，并保存 before/after 摘要；不得由网页拼接任意路径或 Shell 命令。

### 5.6 验证新世界

必须同时证明：

- 官方 REST 返回的新 `worldguid` 非空且不等于旧世界；
- 实际 PalServer 进程来自批准安装根目录；
- DedicatedServerName、运行世界目录和官方 `worldguid` 通过现有身份一致性检查；
- PalDefender `status.connected=true`，版本符合批准组合；
- Native Bridge 协议、game build、mod version、`inventory.probe` 和稳定 `inventory.consume` 通过，且持久化验收没有过期；
- 新世界能完成一次官方 save 并观察到稳定原生快照；
- 数据库里不存在指向新世界的旧 PlayerUID 映射；
- 用测试账户完成只读玩家、位置、背包读取；不在验收阶段发正式商品或入账。

验证通过后，把实际 `worldguid` 与新赛季绑定。绑定是一次性操作；不允许已经存在 worldguid 的赛季改指另一个世界。

### 5.7 开放新赛季

1. 在一个数据库事务中把旧赛季置 `closed`、新赛季置 `open`。
2. 创建本周战备券钱包（余额 0）和首日 rotation。
3. 先只开放身份绑定和只读商城，观察 5 分钟。
4. 完成低风险测试账号结构化发货回执和 Native 幂等扣物/持久化的受控演练；测试物品不得进入正式经济。
5. 先开放购买，再开放资源兑换；记录操作者、时间、版本和健康快照。
6. 广播开服，持续观察 30 分钟。

### 5.8 回滚边界

在 `season_opened` 前可以回滚：关闭新进程，恢复旧 DedicatedServerName 配置，启动旧世界并验证原 worldguid，然后重开旧赛季闸门。回滚也必须形成完整 rollover 事件。

一旦新赛季已经开放且产生任何绑定、订单、扣物或钱包分录，禁止自动回到旧世界。此时只能：

- 关闭经济；
- 修复新世界并继续；或
- 宣告新赛季作废，完整归档后创建另一个全新赛季和世界，并通过账本补偿。

原因是回到旧世界会让游戏物品状态与新赛季数据库交易分叉。

## 6. 灾难恢复

### 6.1 数据库故障

- 自动关闭两个经济闸门，禁止降级为内存钱包或旧 JSONL 钱包。
- Palworld 可继续运行，但网页显示维护状态。
- 恢复 SQLite/WAL 与一致性经济快照后，核对备份时间、outbox、活跃订单和资源兑换，再手动开放。

### 6.2 游戏世界损坏

- 立即关闭经济和登录绑定。
- 停止服务，保留损坏世界和日志。
- 选择最近已验证备份进行离线恢复；现有存档中心 v1 不执行恢复，需要独立审核工具。
- 恢复后若 worldguid/时间点回退，必须对该备份时间之后的所有订单和资源兑换逐笔对账，不能直接开放。

### 6.3 PalDefender 或 Native 版本漂移

- 版本探针不匹配时 fail closed。
- 不通过“忽略版本”临时开放。
- 在开发服重新完成读接口、结构化发货、Native 完整快照扣物、保存/停服/重启/重新登录、uncertain 和崩溃恢复验收后，更新批准组合。

## 7. 监控与告警

最低指标：

- `economy_gate_open{kind}`；
- `orders_total{state}`、`order_delivery_seconds`；
- `extractions_total{state}`、`extraction_settlement_seconds`；
- `wallet_ledger_entries_total{type,currency}`；
- `outbox_oldest_age_seconds`、`outbox_pending`；
- `game_commands_uncertain_total{operation}`；
- `identity_binding_conflicts_total`；
- `world_identity_match`；
- `rcon_local_only_check`、`rcon_commands_uncertain_total`；
- `daily_rotation_age_seconds`；
- `last_verified_game_backup_age_seconds`、`last_database_backup_age_seconds`。

建议立即告警：世界身份不匹配、钱包出现负数（数据库本应阻止）、任何部分扣物、uncertain 超过 5 条、outbox 超过 60 秒、备份超期、版本漂移。

## 8. 禁止操作清单

- 禁止把 Control API 直接映射公网；API Key/RBAC/TOTP 是纵深防御，不能替代回环监听、反向代理路径白名单和防火墙。
- 禁止让浏览器直连 PalDefender、RCON、官方 REST 或 Named Pipe。
- 禁止按昵称发货、绑定或调账。
- 禁止在服务器运行中直接编辑 `.sav`。
- 禁止在归档验证前删除旧世界目录。
- 禁止对 `uncertain` 自动退款、自动重发或自动入账。
- 禁止直接更新钱包余额或删除 ledger。
- 禁止在新赛季产生交易后自动回滚到旧世界。
