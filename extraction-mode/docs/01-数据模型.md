# 01：数据模型

## 1. 数据库选择与约定

当前单机开发服的经济事实来源是 `services/control-api/data/extraction/extraction-commerce.db`：SQLite 使用 WAL、`synchronous=FULL`、唯一事件 ID 和单事务追加，承载账户、钱包、账本、商品、订单、赛季、发货 evidence/receipt、PalDefender command outbox、管理审计和调度。旧经济事件与 PalDefender JSONL 只允许一次性迁移，迁移后不再接收新交易/命令。

下述关系模型同时描述当前 SQLite 领域约束和未来多实例存储目标；只有容量或可用性要求多个 Control API/worker 实例时才评估 PostgreSQL。公告、通知和存档等非经济 JSONL 只保存各自通道 side state，不能承担钱包、限购或经济 outbox 权威。

约定：

- 主键使用 UUID；时间使用 `timestamptz`，统一写 UTC。
- 金额和数量使用 `bigint`，货币精度为 0，禁止浮点数。
- 枚举首版可用受约束的 `text`，迁移时保持向后兼容。
- 所有业务表包含 `created_at`，可变记录包含 `updated_at`。
- 永久账本只追加，不更新或删除；纠错必须新增反向分录。
- JSONB 只保存不可查询的上游快照/证据，关键业务字段必须是普通列。

## 2. 身份与赛季

### `accounts`

| 字段 | 类型 | 约束/说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `display_name` | text | 网页展示名，不作为身份 |

| `created_at` | timestamptz | 非空 |
| `updated_at` | timestamptz | 非空 |

### `platform_identities`

| 字段 | 类型 | 约束/说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `account_id` | uuid | FK `accounts` |
| `provider` | text | 首版仅 `steam` |

| `game_build` | text | 验收通过的游戏版本 |
| `paldefender_version` | text | 验收通过的上游版本 |
| `native_mod_version` | text | 验收通过的 MOD 版本 |

约束：同一时刻只允许一个 `state='open'` 的赛季，使用唯一部分索引实现。

### `season_players`

| 字段 | 类型 | 约束/说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `season_id` | uuid | FK `seasons` |
| `account_id` | uuid | FK `accounts` |
| `platform_identity_id` | uuid | FK `platform_identities` |
| `world_guid` | text | 必须等于赛季世界 |
| `player_uid` | text | 当前世界玩家 UID |
| `last_name` | text | 仅审计展示 |
| `first_seen_at` / `last_seen_at` | timestamptz | 非空 |

唯一约束：

- `UNIQUE(season_id, account_id)`；
- `UNIQUE(world_guid, player_uid)`；
- 绑定时在同一事务内验证身份 `subject == PalDefender.UserId`。

## 3. 钱包与账本

### `currencies`

预置两行：

| `code` | `scope` | `allow_negative` |
| --- | --- | --- |
| `market_credit` | `permanent` | false |
| `supply_ticket` | `season` | false |

### `wallets`

| 字段 | 类型 | 约束/说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `account_id` | uuid | FK |
| `season_id` | uuid | 永久币必须为空，赛季币必须非空 |
| `currency_code` | text | FK `currencies` |
| `balance` | bigint | `CHECK(balance >= 0)` |
| `version` | bigint | 每次写入递增，初始 0 |

PostgreSQL 对 `NULL` 的唯一语义需单独处理：

- 永久钱包唯一部分索引：`UNIQUE(account_id, currency_code) WHERE season_id IS NULL`；
- 赛季钱包：`UNIQUE(account_id, season_id, currency_code) WHERE season_id IS NOT NULL`。

### `wallet_ledger`

| 字段 | 类型 | 约束/说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `wallet_id` | uuid | FK |
| `entry_type` | text | `shop_debit/shop_reversal/extraction_credit/admin_adjustment/season_reward/expiry` |
| `amount` | bigint | 有符号；扣款为负数 |
| `balance_after` | bigint | 写入后的权威余额 |
| `reference_type` | text | `order/extraction/season/admin` |
| `reference_id` | uuid | 对应领域记录 |
| `idempotency_key` | text | 业务幂等键 |
| `reason` | text | 可审计原因 |
| `actor_type` / `actor_id` | text | `player/system/operator` |
| `metadata` | jsonb | 价格版本、IP、上游命令等证据 |
| `created_at` | timestamptz | 非空 |

唯一约束：`UNIQUE(wallet_id, idempotency_key)`，以及对系统分录使用 `UNIQUE(wallet_id, entry_type, reference_type, reference_id)`。这保证崩溃恢复时同一资源兑换只能入账一次。

写钱包的唯一合法过程：

1. 开启数据库事务；
2. `SELECT ... FOR UPDATE` 锁定 wallet；
3. 验证币种范围、赛季和余额；
4. 更新 `balance`、`version`；
5. 插入 ledger；
6. 插入 outbox；
7. 提交。

禁止绕过账本直接修正余额。

## 4. 商城

### `shop_products`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `sku` | text | 全局唯一、稳定 |
| `name` | text | 中文展示名 |
| `grant_type` | text | MVP 仅 `item` |
| `item_id` | text | Palworld 内部 ItemID |
| `grant_quantity` | bigint | 单件 offer 发放量 |
| `enabled` | boolean | 总开关 |
| `metadata` | jsonb | 分类、图标和说明 |

### `item_economy_rules`

该表是资源兑换回收白名单，不等同于商城商品。字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `item_id` | text | PK，精确区分大小写的 Palworld ItemID |
| `display_name` | text | 中文展示名 |
| `sellable` | boolean | 默认 false，只有显式审核后才 true |
| `base_sell_price` | bigint | 回收基价；不可售时为 0 |
| `allowed_containers` | text[] | MVP 只能是 `Items/Food/DropSlot` 子集 |
| `consume_adapter` | text | MVP 固定 `native-inventory-consume-v1` |
| `consume_validated` | boolean | 当前固定版本上是否完成真实扣除、保存、停服、重启、重登与完整回执验证 |
| `validation_hash` | text | 测试证据与版本组合摘要 |
| `validated_at` | timestamptz | 可空 |
| `enabled` | boolean | 紧急关闭 |

只有 `sellable=true AND consume_validated=true AND enabled=true` 的 ItemID 能进入报价和 Native `inventory.consume`。游戏、UE4SS、Native MOD/协议或快照 schema 变化会让 validation 失效；目录外 ItemID 默认拒绝。商品买入与资源兑换卖出共用 ItemID 时，发布检查仍强制卖价不高于最低买价的 50%。

### `shop_rotations`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `season_id` | uuid | FK |
| `business_date` | date | `Asia/Shanghai` 业务日 |
| `state` | text | `draft/published/expired` |
| `rules_version` | text | 生成器版本 |
| `content_hash` | text | 防止同日内容漂移 |
| `published_at` | timestamptz | 可空 |

唯一约束：`UNIQUE(season_id, business_date)`，使日刷新可安全重复执行。

### `shop_offers`

| 字段 | 类型 | 说明 |
| --- | --- | --- |

| `rotation_id` | uuid | FK |
| `product_id` | uuid | FK |
| `currency_code` | text | `market_credit` 或 `supply_ticket` |
| `unit_price` | bigint | 发布后不可变 |
| `per_account_limit` | bigint | 0 表示不开放，不使用 NULL 表示无限；无限另设布尔字段 |
| `available_from` / `available_until` | timestamptz | 非空 |
| `enabled` | boolean | 紧急下架开关 |

### `orders`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `account_id` / `season_id` | uuid | FK |
| `season_player_id` | uuid | 发货目标快照 |
| `currency_code` | text | 非空 |
| `total_amount` | bigint | 非空、正数 |
| `state` | text | 见交易状态机 |
| `client_idempotency_key` | text | 玩家请求键 |
| `request_hash` | text | 规范化请求摘要 |
| `player_uid_snapshot` / `user_id_snapshot` | text | 发货审计 |

### `order_lines`

保存购买时不可变快照：`offer_id`、`sku`、`item_id`、购买件数、每件发放数量、单价、总价。后续商品改价不能影响旧订单。

### `deliveries`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `order_id` | uuid | FK，MVP 唯一 |
| `attempt` | integer | MVP 固定 1；人工补偿创建新 delivery，不覆盖旧记录 |

## 5. 资源兑换

### `extraction_zones`

字段：`id`、`season_id`、`name`、`map_x`、`map_y`、`radius`、`opens_at`、`closes_at`、`enabled`、`config_version`。首版只支持圆形区域，位置计算始终在服务端完成。

### `inventory_snapshots`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | PK |
| `season_player_id` | uuid | FK |
| `source` | text | `paldefender-rest-before/rcon-rest-after/native-post-consume` |
| `revision` | text | 上游有则保存 |
| `snapshot_hash` | text | 对规范 JSON 做 SHA-256 |
| `payload` | jsonb | 六类容器原始证据 |
| `captured_at` | timestamptz | 非空 |

### `extractions`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | PK，同时是 quote ID |
| `account_id` / `season_id` / `season_player_id` | uuid | FK |
| `zone_id` | uuid | FK |
| `state` | text | 见资源兑换状态机 |
| `currency_code` | text | MVP 固定 `supply_ticket` |
| `quoted_amount` | bigint | 非负 |
| `quote_expires_at` | timestamptz | 默认 30 秒 |
| `before_snapshot_id` / `after_snapshot_id` | uuid | FK |
| `snapshot_hash` | text | 确认时匹配 |
| `client_idempotency_key` / `request_hash` | text | 幂等 |
| `consume_command_id` | uuid | Pal Control 内部消费命令 ID，不等同于 RCON ACK |
| `failure_code` / `failure_message` | text | 可空 |
| `created_at` / `updated_at` | timestamptz | 非空 |

唯一部分索引保证同一 `season_player_id` 仅有一个 `quoted/consume_queued/consuming/consume_uncertain/consumed/crediting` 状态的活动资源兑换。

### `extraction_lines`

每行保存：`container_kind`、`container_id`、`slot_id`、`item_id`、`quantity`、`unit_sell_price`、`line_amount`、`price_rule_version`。确认后不可修改。

### `inventory_consumptions`

保存扣物命令：`extraction_id`、`adapter`（正式路径固定为 `native-inventory-consume`；显式隔离的 Development 诊断可记录 `development-rcon`）、`state`、`before_snapshot_id`、`after_snapshot_id`、`idempotency_key`、`dispatched_at`、`completed_at`、`transport_result`、`evidence`。`UNIQUE(extraction_id)` 和 `UNIQUE(idempotency_key)`。

`idempotency_key` 约束本地只能生成一条消费记录。Native 在同一游戏 Tick 验证冻结的完整 before snapshot、原子扣除全部报价行，并返回逐槽位、逐行、完整 after snapshot 与聚合回读证据；所有证据完全一致时才允许 `state=consumed`。一旦请求可能已进入游戏线程但结果无法证明，记录必须进入 `uncertain`，恢复 worker 不得自动重发。Development RCON 诊断不属于正式结算能力，也不能作为生产降级路径。

## 6. 任务、调度和 Outbox

### `scheduled_runs`

记录日刷新和周换档：`job_type`、`schedule_key`、`state`、`started_at`、`completed_at`、`result`。唯一约束 `UNIQUE(job_type, schedule_key)`，例如 `daily-refresh:2026-07-13`。

### PalDefender 发货 outbox

`paldefender_commands` 是单机商城发货命令投影，字段包括 `command_id`、`server_id`、`upstream_path`、`request_body`、`idempotency_key`、`request_hash`、`state`、`available_at`、`lease_owner/lease_until`、`attempt_count/failure_count`、`dispatched_at/completed_at`、脱敏结果与 `dead_lettered_at`。数据库强制 `UNIQUE(server_id, idempotency_key)`，同 key 不同 hash 在进入游戏写入前冲突。

`paldefender_command_events` 与 accepted/lease/dispatch/terminal/dead-letter 转移在同一 SQLite 事务写入；UPDATE/DELETE trigger 使事件不可变，命令 trigger 禁止重开终态或修改请求 envelope。worker 使用 `BEGIN IMMEDIATE` 和条件 UPDATE 获取 30 秒租约；当前只支持一个 Control API/worker 实例，多实例不是 MVP。

购买事务先在 `extraction_events` 中持久化订单、扣款、ledger、限购和 pending delivery。发货 worker 随后以确定性 delivery key 在同一个 `extraction-commerce.db` 的独立事务注册 receipt request 和 PalDefender command。因此“扣款 + command accepted”不是一个事务；崩溃恢复会从 pending delivery/receipt 查找同一个 server+key 后补齐，不能生成新 key。若未来改为多实例，再把这条边界合并为购买事务内的通用 outbox，而不是假定当前已经原子提交。

Outbox 只能重试数据库内幂等步骤和“确定尚未进入 `dispatched`”的工作。游戏命令一旦进入 `dispatched/uncertain`，只有已持久化成功回执允许恢复后继续唯一入账；否则只能对账，不能再次发送。连续 5 次派发前内部故障进入 dead-letter，并触发购买熔断。

### `audit_events`

面向管理员查询的追加式事件：操作者、IP、动作、目标、before/after、reason、requestId、关联订单/资源兑换/赛季、游戏、PalDefender、Native 协议/game build/MOD 版本与可选 RCON 诊断版本。敏感 token、RCON 密码和完整认证 Cookie 永不写入。

## 7. 关键不变量

数据库迁移与测试必须证明：

1. 任一钱包余额永不小于 0。
2. 一个客户端幂等键不能创建两个订单或两个资源兑换。
3. 一个订单最多有一笔最终商城扣款；退款必须是独立反向分录。
4. 一个资源兑换最多有一笔 `extraction_credit`。
5. `extraction_credit` 只能引用状态至少达到 `consumed` 的资源兑换。
6. `supply_ticket` 钱包必须引用赛季；`market_credit` 钱包不得引用赛季。
7. 新世界不能复用旧世界 `PlayerUID` 映射。
8. `uncertain` 状态不能由普通 worker 自动跳转为重试；只能由对账证据转为终态。
9. 发布后的 offer 和 extraction line 价格快照不可修改。
10. 关闭赛季后不能创建新订单、报价或绑定。
11. 生产资源消费只能选择已通过版本锁定和能力探针的 Native adapter；PublicSteam、非显式开发模式或 Native 必选配置下不得选择 RCON。
12. 只有 Native 返回的逐槽位、逐行、完整快照和聚合回读全部精确证明报价数量已减少，消费记录才能进入 `consumed`；任何不确定请求不得自动重发。
